Introduction

KGDB is a source level debugger for linux kernel. It is used along with GDB to debug linux kernel. Kernel developers can debug a kernel similar to application programs with use of KGDB. It makes it possible to place breakpoints in kernel code, step through the code and examine variables.

Requirements

Two systems are required for using KGDB. The development system, which has the target linux source tree, kernel image and system map file, runs GDB. The target system runs the linux kernel to be debugged.

For remote debugging over serial interface a serial connection is required between the development and the test systems. Serial connection is usually established by connecting the serial ports on both systems with a serial cable. For Broadcom Set-Top Box target system a DB-9 serial cable is needed, however, some other platforms may require a null modem cable.

On the development system the appropriate toolchain has to be set up in order to bring up the correct version of GDB. The command below will display whether GDB is properly installed.

[linux prompt] which mipsel-linux-gdb <CR>

Enabling KGDB for Kernel Debugging

KGDB requires a special kernel build.  Example for 7405b0:

Normal kernel build: make vmlinuz-initrd-7405b0 install

KGDB kernel build: make vmlinuz-initrd-7405b0-kgdb install

Adding the "-kgdb" option changes numerous kernel configuration options automatically, and enables most of the debugging features (e.g. spinlock and malloc debugging).

To start the kernel debugging process, reboot the target system with the following CFE command line.

<CFE> boot elf <SERVER_NAME>:vmlinuz-initrd-7405b0-kgdb kgdboc=ttyS1,115200 kgdbwait<CR>

Where <SERVER_NAME> is the server name that the kernel image resides. e.g. stb-irva-01. The kgdbwait option on the command line tells the kernel to pause startup until it detects that gdb has connected; this is completely optional.  "kgdboc=ttyS1,115200" tells KGDB to attach to UARTB and set the line rate to 115200bps.

KGDB over Ethernet is not supported.

For kernel module debugging, it is recommended that you compile all drivers statically into the kernel image instead of building them separately as modules.  Any binary modules you are using (e.g. bcmdriver.ko) will need to be rebuilt against the new kernel headers.

Note that UARTB/UARTC may be disabled on some boards due to pin sharing.  On 97405, SW2801-7 must be in the OFF position, and any MII daughtercards must be removed.

 

 

Connecting to KGDB for Kernel Debugging

 

If you built a single target as recommended in the previous section (e.g. vmlinuz-initrd-7405b0), the stblinux-2.6.*/vmlinux will contain the uncompressed ELF image for your new kernel.  To load that image up in the host gdb:

$ cd ../stblinux-2.6.31
$ mipsel-linux-gdb vmlinux

If you booted with the kgdbwait option, the target should be waiting here:


Waiting for connection from remote gdb...

Then connect to the target machine from GDB using target remote command. This command needs the serial line path to be specified as an argument. At this point GDB and the KGDB stub will be connected and GDB will have a control of the target kernel:

(gdb) set remotebaud 0x1c200
(gdb) target remote /dev/ttyS0

Note: /dev/ttyS0 is the local port on the PC, not the STB.

Note 2: 0x1c200 = 115200 decimal.

At this point gdb should be connected to the kgdb server on the target, and it will show that the kernel is stopped.  You may resume the kernel boot:

(gdb) continue

You can now use GDB commands for inserting breakpoints, printing values of variables, breaking and continuing execution.

If a kernel panic situation occurs, instead of declaring a panic, the kernel will first give control to GDB so the cause of system panic can be analyzed.

Using GDB For Kernel Debbuging

Usage of GDB kernel debugging is similar to that for debugging application processes. Please refer to GDB documentation for a list of GDB commands and explanations of their usage. GDB documentation can be accessed by the command info gdb. GDB has online help also. The GDB command help gives a short description of a command name or topic name given as an argument.

 

Invoke GDB session

In order to break into the debugger, do not hit ^C in gdb.  Instead, in the target's console window, send <BREAK> g :

Minicom - ^A f g
Tera Term - 1) Control -> Send break; 2) Hit 'g'

Note that <BREAK> on the STB works like Alt-SysRq on a PC.  If you send <BREAK> ? , the kernel will print a list of other debugging commands that are available.

Terminating/Resuming GDB session

Debugging session can be terminated by the GDB command detach, and a new debugging session can be started by the GDB command target remote <serial interface> again.

Test machine reboot

If a machine is rebooted, it is recommended that you terminate or kill GDB before the target kernel starts again.

Continuing kernel execution

After a breakpoint is encountered, to continue kernel execution, use GDB command continue. Kernel execution will continue after this point. GDB will not expect a user command till the user either breaks execution using <BREAK> g, or KGDB stub gives control to it because of another breakpoint or some other exceptions occur.

Breakpoints

Use GDB break command to stop kernel execution at a function or a source code line. The command accepts a function name or a source code file name appended with a : and a line number as an argument.   Example:

(gdb) break do_fork
(gdb) continue

Run a program in order to trigger the breakpoint:

# busybox

After entering gdb, disable the breakpoint and resume execution:

(gdb) disable
(gdb) continue

Stepping through code

To step a code statement, use GDB command step. This command will run the kernel for one statement and again hand over the control to GDB. This command will step into function calls also. If you want to skip stepping into function calls, use GDB command next.

Stack Trace

Once GDB is in command mode, you can look at the stack trace using backtrace command. This command shows a list of function calls starting from the function where a system call entered the kernel. For each function, it prints the source code file name and line number where next function was called and for the innermost function, where execution stopped. It also prints argument values to these functions.

An example of stack trace is shown as follows:

(gdb) backtrace

#1  0xc0188b6c in gdb_interrupt (irq=3, dev_id=0x0, regs=0xc02c9f9c)
    at gdbserial.c:143
#2  0xc0108809 in handle_IRQ_event (irq=3, regs=0xc02c9f9c, action=0xc12fd200)
    at irq.c:451
#3  0xc0108a0d in do_IRQ (regs={ebx = -1072672288, ecx = 0, edx = -1070825472,
      esi = -1070825472, edi = -1072672288, ebp = -1070817328, eax = 0,
      xds = -1072693224, xes = -1072693224, orig_eax = -253,
      eip = -1072672241, xcs = 16, eflags = 582, esp = -1070817308,
